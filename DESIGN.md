# DESIGN.md

目标网站：https://developers.openai.com/api/docs
生成日期：2026-06-08
范围：基于 Chrome 中可访问页面的视觉、响应式表现与样式信息生成，用于指导后续同风格页面实现。

## 设计原则

- 以文档阅读效率为第一目标：界面应安静、留白充足、层级清晰，避免营销页式的大面积装饰。
- 主要视觉语言是黑白灰加少量蓝色链接/焦点色；彩色只用于内容图片、代码高亮和状态语义。
- 组件应低阴影、低饱和、细边框，以圆角和浅灰底区分层级。
- 文档页优先使用固定顶栏、桌面侧边导航和宽内容区；移动端收敛为顶部品牌栏与单列内容。
- 代码、参数、表格和卡片是核心内容载体，必须保证扫描速度和复制/阅读体验。

## 暗色模式

OpenAI Developers 支持 `data-theme="dark"` / `.dark` 暗色主题。后续实现不能只做浅色模式；暗色模式应沿用同一套语义 token，通过根节点主题属性切换实际色值。

| 用途 | 建议 token | 浅色值 | 暗色值 | 使用规则 |
| --- | --- | --- | --- | --- |
| 页面背景 | `background` / `gray-0` | `#fff` | `#0d0d0d` | 页面级背景随主题反转；暗色模式避免使用纯黑以外的大面积彩色底。 |
| 基础表面 | `surface-muted` / `gray-50` | `#f9f9f9` | `#131313` | 用于代码面板、搜索框、浅提示区；暗色下需要比页面背景高一级。 |
| 次级表面 | `surface-tertiary` / `gray-75` | `#f3f3f3` | `#161616` | 用于 hover、菜单项、局部面板，不要在暗色下直接复用浅灰。 |
| 激活表面 | `surface-active` / `gray-100` | `#ededed` | `#181818` | 用于当前导航项、segmented 选中态和浅按钮 active。 |
| 主文本 | `text-primary` / `gray-1000` | `#0d0d0d` | `#fff` | 用于标题、主导航选中项、关键数值；暗色下保持最高对比。 |
| 默认文本 | `text-default` / `gray-750` | `#282828` | `#b9b9b9` | 用于正文和说明文字；暗色下不要使用纯白作为所有正文。 |
| 次级文本 | `text-secondary` / `gray-550` | `#5d5d5d` | `#767676` | 用于描述、占位、辅助信息；暗色下需通过实际屏幕检查可读性。 |
| 弱化文本 | `text-tertiary` / `gray-600` | `#8f8f8f` | `#8f8f8f` | 用于分组标签和低优先级元信息，避免低于可读对比。 |
| 细边框 | `border-subtle` | `color-mix(in oklab, #0d0d0d 5%, transparent)` | `color-mix(in oklab, #fff 8%, transparent)` | 暗色模式边框使用白色透明度，不使用实心亮灰线。 |
| 常规边框 | `border-default` | `#ededed` | `#303030` | 用于卡片、表格、分隔线；暗色下边框应弱于文本但高于背景。 |
| 链接与焦点 | `link-primary` | `#0169cc` | `#66b5ff` | 暗色链接需要提高亮度；focus ring 保持蓝色但降低外发光感。 |
| 链接 hover | `link-primary-hover` | `#013566` | `#99ceff` | hover 通过亮度变化表达，不额外铺大面积蓝色背景。 |
| 细描边阴影 | `shadow-hairline-color` | `#00000014` | `#ffffff1a` | 暗色下阴影主要作为内描边/发丝线，层级更多依赖边框和表面差异。 |
| 代码背景 | `codeblock-background` | `#fcfcfc` | `#131313` | 代码区使用独立表面；语法高亮在暗色下应提升蓝、紫、绿的亮度。 |

- 主题切换应同时设置 `color-scheme: light dark` 或当前模式对应的 `color-scheme`，保证浏览器原生控件、滚动条和表单状态一致。
- 单色 logo 和图标可在暗色模式使用 `invert` 或主题专用资源；内容图片、博客封面和产品截图默认不反相。
- 组件不要维护两套独立样式；按钮、导航、卡片、代码块、表格都应消费语义 token，由主题层决定色值。
- 暗色模式中减少阴影表达，优先使用 `#131313`、`#161616`、`#181818`、`#303030` 等灰阶差异建立层级。
- 移动端浮动入口、菜单和搜索 overlay 也必须适配暗色背景，不能保留浅色卡片或黑字。

## 组件规范

### 按钮

- 主按钮使用黑色胶囊外观，白色文字，约 `36px` 高，`14px` 字号，`500` 字重，水平 padding 约 `16px`。
- 次按钮使用浅灰表面或透明底，深灰文字，同样使用胶囊圆角，避免彩色填充。
- 顶栏 CTA 可使用黑色胶囊并带右上箭头图标；图标与文字间距约 `6px`。
- 图标按钮尺寸保持 `36-38px`，图标本体 `16-20px`，颜色使用 `text-secondary`，hover 变为主文本色。
- Segmented/chip 控件使用 `14px / 20px`，`6px 14px` padding，`1px` 浅边框，选中态填充 `#ededed`。
- 禁止把多个主要 CTA 同时做成高对比黑色；同一区块只保留一个主操作。

### 导航

- 顶栏固定在页面顶部，高 `64px`，白底，底部使用极浅边框；品牌在左，产品导航居中，搜索和 CTA 在右。
- 桌面文档页使用左侧导航，分组标题为 `14px / 20px / 600`，导航项为 `14px / 20px`。
- 当前导航项使用浅灰背景和圆角，不使用粗重左边线或高饱和色块。
- 移动端顶栏只保留品牌、搜索图标和菜单图标；主导航与侧栏内容收敛到菜单或内容下方。
- 导航 hover 只做轻量颜色/背景变化，过渡时间约 `150ms`。

### 卡片与容器

- 普通入口卡片使用白底、`1px` 浅边框、`10-12px` 圆角、`16px` padding；hover 可轻微改变边框或背景，不加明显阴影。
- 大型 hero/代码复合容器使用 `#f9f9f9` 背景、`16px` 圆角，内部左右分栏；移动端改为垂直堆叠。
- 提示块使用白底、`1px` 边框、`12px` 圆角，标题 `14px / 20px / 600`，正文 `14px / 24px`。
- 卡片内标题、描述、链接动作按 8-12px 递进间距排列；不要在卡片内再嵌套卡片。
- 图片型内容卡片在桌面可使用 `6-8px` 圆角；移动端允许图片全宽无圆角，文字部分使用 `16px` 左右边距。

### 表单

- 搜索输入使用胶囊/大圆角，背景为黑色 8% 左右的浅灰混合，边框透明或极浅。
- 大搜索框可用 `18px` 字号、`16px 56px 16px 24px` padding；顶栏搜索框使用 `14px` 字号和 `38px` 高度。
- 输入 focus 使用蓝色 ring `#0169cc` 或更明显的浅灰边框，不使用厚重外发光。
- 表单标签使用 `14px`，说明文本使用 `text-secondary`，错误文本使用 `danger`。

### 内容区块

- 文档正文以标题、短段落、代码块、表格和卡片入口组织，避免大段未分隔文字。
- API reference 类型页面使用更紧凑的信息密度，参数、端点、示例代码应并列或紧邻出现。
- Pricing/参数表格应保持细线、紧凑行高、浅灰分组，不使用重色表头。
- 博客/资源页可使用更大留白和图片首图，但仍遵循黑白灰 UI 框架。
- 浮动辅助入口可使用黑色胶囊按钮，固定在右下角，移动端保持足够触控尺寸。

### 代码块

- 代码块使用浅灰背景 `#f9f9f9` 或 `--codeblock-background-color: #fcfcfc`，`12px` 圆角。
- 代码字体使用 `font-mono`，正文代码字号约 `14px`，行号使用更弱的灰色。
- 代码头部的语言选择或复制按钮应小而清晰，图标为线性风格。
- Inline code 使用浅灰透明底、内描边、`6px` 圆角和 `2px 8px` padding。

## 图像与图标风格

- UI 图标使用单色线性图标，尺寸以 `16px` 和 `20px` 为主，颜色继承文本灰阶。
- 导航和操作图标不使用彩色填充；只在 hover/focus 时提高对比。
- 内容图片可以使用高饱和、柔和渐变、抽象技术图形或产品场景，但应服务于内容主题。
- 博客卡片图片桌面端可保留圆角；移动端可全宽铺开，形成更强的内容节奏。
- 不使用独立装饰光球、复杂背景纹理或大面积品牌渐变作为 UI 背景。

## 交互与动效

- 常规 hover/focus/active 过渡使用 `150ms cubic-bezier(0.4, 0, 0.2, 1)`。
- 顶栏或布局宽度变化可使用约 `300ms` 过渡。
- Hover 以颜色、边框、背景轻微变化为主；不要使用大位移、弹跳或强阴影。
- 可点击卡片 hover 时只做浅背景或边框变化，保持文档界面的稳定感。
- Focus 状态必须可见，优先使用蓝色 ring `#0169cc`。

## 响应式规则

- 桌面端使用固定顶栏 + 左侧导航 + 主内容区域；左侧导航宽约 `240px`，主内容从顶栏下方开始。
- 桌面主内容可在 `72rem` 内组织大卡片和两列卡片网格，宽卡片最大约 `1120px`。
- 移动端顶栏保持 `64px` 高，内容从顶栏下方开始，左侧导航隐藏或移动到菜单/内容后部。
- 移动端主内容单列排列，宽度占满视口，文字和卡片内容保留 `16px` 左右安全边距。
- 移动端图片和媒体可全宽显示；卡片网格必须改为单列，不横向滚动。
- 移动端按钮组可横向滚动，但主要 CTA 应可完整显示；避免把多个胶囊按钮挤在一行导致文字截断。
- `380px` 是极小屏断点参考；在 `390-430px` 宽度下仍要保证导航、搜索和浮动按钮不遮挡正文。

## Do / Don't

| Do | Don't |
| --- | --- |
| 使用黑白灰作为 UI 主体，蓝色只承担链接和焦点角色。 | 把蓝色或彩色渐变当成大面积品牌背景。 |
| 用浅灰表面、细边框、圆角表达层级。 | 用重阴影、厚边框或强分割线制造层级。 |
| 保持文档页高密度但有节奏，标题、说明、代码、表格紧密配合。 | 把文档页做成营销落地页，使用超大 hero 和装饰性排版。 |
| 桌面端保留侧栏导航，移动端收敛为单列和菜单。 | 在移动端保留桌面侧栏或制造横向滚动。 |
| 代码块、inline code、参数表格使用一致的 mono 字体和浅灰底。 | 让代码内容与普通正文没有视觉区分。 |
| 图标保持线性、单色、小尺寸。 | 混用填充彩色图标、插画图标和不同线宽。 |
| 卡片 hover 只做轻微反馈。 | 对文档卡片使用夸张缩放、发光或复杂动画。 |

## 后续实现注意事项

- 优先建立 token 层：颜色、字体、圆角、间距、边框和过渡都应集中定义，组件只消费语义 token。
- 如果项目没有 `OpenAI Sans`，先使用系统 sans-serif 作为 fallback，并保持 `16px / 24px` 的正文节奏。
- 文档型页面应优先实现 `Header`、`SidebarNav`、`DocLayout`、`CardLink`、`CodeBlock`、`InlineCode`、`SearchInput`、`PillButton` 和 `DataTable`。
- 新增页面时先确定内容密度：文档/参考页走紧凑布局，博客/资源页可以增加留白和图片面积。
- 组件圆角不要随意扩大；普通卡片控制在 `10-12px`，大型面板控制在 `16px`，胶囊控件才使用 `9999px`。
- 不要把临时采集记录写入实现文档；后续实现只应依赖本文件中的 token 和组件规则。
